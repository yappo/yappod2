# Wikipedia検索 Web UIのUX設計

この文書は画面の装飾ではなく、利用者が迷わず検索、比較、原典確認、文書登録を完了するための
情報設計を定義します。Stage 3の検索と文書登録に加え、Stage 4で質問と引用付き回答を実装します。

## 利用者の主要タスク

1. 調べたい語句を入力し、検索を開始する。
2. title、出典URL、snippetを上から比較する。
3. 適切な結果をWikipediaの原典で読む。
4. 必要な文書を補助画面から登録し、登録した内容を再検索する。
5. 質問し、回答と引用元を同じ読む順序で確認する。

検索結果のscore、generation、document IDは診断には必要ですが、通常の比較には不要です。そのため
結果ごとの詳細領域へ置き、title、出典、snippetの読む順序を妨げないようにします。

## 画面構成と優先順位

- 共通header: サービス名、検索・質問・文書登録の切り替え、daemon状態。
- 検索: query入力、検索実行、件数、結果一覧、次ページ取得の順。
- 文書登録: ID、title、URL、本文、登録実行、成功後の再検索の順。
- 質問: 質問、根拠に基づく回答、参照資料の順。チャット形式にはしない。

desktopでもsidebarを置かず、本文は読みやすい幅の一列にします。mobileでも要素の順序は変えず、
検索buttonと入力欄だけを縦に並べます。

## 状態と復旧経路

| 状態 | 表示 | 次に取れる行動 |
|---|---|---|
| 初期 | 入力例と短い説明 | queryを入力する |
| loading | 操作位置の近くに進行表示 | 完了を待つ。入力内容は保持する |
| 0件 | 「一致する記事がありません」 | 語句を短くする、表記を変える |
| 入力不正 | 対象field直下の理由 | 入力を修正して再実行する |
| daemon停止 | headerと本文に接続不能を表示 | daemon起動後に再接続する |
| 検索失敗 | 検索条件を保持したerror | 同じ条件で再試行する |
| 登録失敗 | formを保持したerror | token/daemon/入力を確認し再試行する |
| 登録成功 | generationと再検索button | 登録文書を検索する |
| LLM未設定 | retrievalは利用可能と説明 | contextと参照資料を確認する |
| LLM失敗 | 取得済み資料を残す | 同じ質問を再実行する、参照資料だけ確認する |
| 不正citation | 回答を非表示にして理由を示す | 取得済み参照資料を確認する |

## Heuristic review

- 検索開始はquery入力とbuttonの1組だけで理解できる。
- title、URL、snippetが同じ縦方向に並び、視線を左右へ往復させない。
- 原典移動はtitleのlinkで1操作、詳細scoreは必要時だけ1操作で展開できる。
- 文書登録は独立navigationにあり、通常検索中に管理操作を誤実行しない。
- RAG回答の`[n]`から対応する参照資料へ1操作で移動できる。
- loading、0件、接続不能、入力不正を同じメッセージで代用しない。
- keyboard focus、44px以上の主要操作領域、本文16px、色以外の状態表現を実装条件とする。

## 視覚設計方針

白を本文背景、チャコールを文字、抑制した青緑をlinkと主要操作に使います。情報階層は主に文字サイズ、
太さ、余白、罫線で作ります。巨大hero、全面カード、pillの多用、gradient、glassmorphism、チャット吹き出し、
装飾画像、意味のないdashboard sidebarは使用しません。
