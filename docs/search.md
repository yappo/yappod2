# 検索の仕組みと指定方法

Yappod2は、語句による語彙検索、事前計算したベクトルによる類似検索、両方の順位を統合する複合検索を提供します。検索結果は文書単位または本文断片単位で返せます。外部APIのリクエストとレスポンスの全項目は[`yappod_front` API](yappod-front-api.md)を参照してください。

## 共通する検索処理

すべての検索は、一つのマニフェスト世代から開いたスナップショットに対して実行します。検索中に更新が公開されても、処理途中で新旧セグメントを混ぜません。

複数セグメントに同じ文書IDがある場合は新しいセグメントを優先します。新しい削除標識がある文書は候補から除きます。本文断片の候補も親文書が現在有効かを確認します。

HTTP APIのデフォルトは次のとおりです。表内でバッククォートに囲んだ文字列は、JSONへそのまま指定する値です。

| キー | データ型 | 入力可能値 | デフォルト値 | 必須 | 説明 |
|---|---|---|---|---|---|
| `mode` | 文字列 | `lexical`、`vector`、`hybrid` | `hybrid` | 任意 | 使用する検索方式を指定します。 |
| `operator` | 文字列 | `or`、`and` | `or` | 任意 | 語彙検索で、検索語のいずれかへの一致またはすべてへの一致を選びます。 |
| `phrase` | 真偽値 | `true`、`false` | `false` | 任意 | `true`では検索語が同じ順序で連続する候補だけを残します。 |
| `limit` | 整数 | 1〜100 | `20` | 任意 | 返す検索結果または採用する本文断片の最大件数です。 |

内部では最低100件の候補を調べます。カーソルを使って続きを取得する場合は、必要な開始位置まで候補数を増やします。この候補数をリクエストから直接指定することはできません。

CLI `search`だけは`limit = 10`をデフォルトとしてリクエストへ入れます。

## 語彙検索

検索文を索引と同じトークナイザーで正規化・分割し、題名、本文、本文断片の転置索引を検索します。

### BM25F

デフォルトは次のとおりです。

| 係数 | 値 |
|---|---:|
| `k1` | 1.2 |
| `b` | 0.75 |
| 題名の重み | 2.0 |
| 本文の重み | 1.0 |
| 本文断片の重み | 1.0 |

`k1`は、同じ語が一つの文書へ繰り返し現れたときに、スコアがどこまで増え続けるかを調整します。値を大きくすると
出現回数の差がスコアへ反映されやすくなり、小さくすると最初の数回でスコアが飽和しやすくなります。`b`は文書の長さを
補正する強さです。0では長さを補正せず、1へ近づけるほど、長い文書に語が現れやすい影響を強く差し引きます。

各フィールドの語句頻度を、そのフィールドの平均長と対象フィールドの長さで正規化し、重みを掛けて合算します。逆文書頻度はセグメント内の対象数と文書頻度から計算します。題名に同じ語がある文書は、同条件なら本文だけにある文書より高くなります。現在の`k1`、`b`、各フィールドの重みは固定値で、TOMLや検索リクエストから変更できません。

実装が検索語ごとに計算する式は次のとおりです。

```text
field_tf = Σ field_weight × term_frequency
                         ÷ (1 - b + b × field_length ÷ average_field_length)

idf = log(1 + (object_count - document_frequency + 0.5)
              ÷ (document_frequency + 0.5))

score = idf × field_tf × (k1 + 1) ÷ (k1 + field_tf)
```

`term_frequency`は対象フィールド内の出現回数、`field_length`は対象フィールドの検索語数、`average_field_length`は
同じセグメントにある同種フィールドの平均検索語数です。`object_count`は文書検索なら文書数、本文断片検索なら本文断片数、
`document_frequency`はその検索語を含む文書または本文断片の数です。複数の検索語がある場合は、各検索語のスコアを加算します。

`terms.yap2`から検索語を二分探索し、`postings.yap2`の128件単位のブロックを読みます。ブロックに保存した最大語句頻度と最小フィールド長から上限スコアを求め、現在の上位`k`件へ届かないブロックをBlock-Max WANDで飛ばします。最終スコアは転置リストの実値から計算します。

### `operator`

- `or`は、一つ以上の検索トークンが現れるオブジェクトを候補にします。
- `and`は、すべての検索トークンが同じオブジェクトに現れることを要求します。

正規化後に同じトークンが複数回現れる検索文では、語彙状態を重複して作らず処理します。

### `phrase`

`phrase = true`では、すべての検索トークンが同じフィールド内で検索文の順に連続して現れることを位置情報から確認します。フレーズを有効にすると、実質的に全トークン一致も必要です。題名から本文へまたがる一致や、別の本文断片へまたがる一致はフレーズになりません。

## ベクトル検索

ベクトル検索は検索文から埋め込みを生成しません。呼び出し側が索引と同じモデル、前処理、`dimensions`で検索ベクトルを作ります。search-webでは`[embedding]`が設定されている場合にsearch-webサーバーが担当します。

リクエストのベクトルは、要素数が索引`dimensions`と同じで、全要素がfloat32へ変換できる有限値でなければなりません。索引が`vector.enabled = false`の場合、ベクトル検索または複合検索のリクエストは不正です。

### 距離尺度とscore

| metric | score | 並び順 |
|---|---|---|
| `cosine` | 二つのベクトルのcosine類似度 | 大きい順です。0 normを含む場合は0です。 |
| `dot` | 内積 | 大きい順です。 |
| `l2` | 二乗Euclidean distanceの負値 | 0に近い大きな値ほど近い結果です。 |

ベクトルを1件ずつすべて比較すると、文書数に比例して検索時間が増えます。そのため通常の索引作成では、近いベクトルの
候補を少ない比較回数で探すためのUSearch HNSWデータを`vectors.usearch`へ保存します。この方式は、すべてのベクトルを
比較せずに近い候補を推定する「近似最近傍検索」です。

検索時は`vectors.usearch`から有望な候補を取得し、利用者へ返すスコアは`vectors.yap2`に保存した元のfloat32ベクトルから
Yappod2が再計算します。`vectors.usearch`がないデータを内部処理へ渡した場合は全ベクトルを比較しますが、通常の
`yappo_makeindex build`は二つのファイルを一緒に作成し、件数と設定が一致することを検証します。近似検索では、真に近い
ベクトルが候補へ入らず検索結果から漏れる可能性があります。

## 複合検索

語彙検索とベクトルの生スコアは尺度が異なるため、直接加算しません。それぞれの候補をスコア順に並べ、Reciprocal Rank Fusionで統合します。

```text
寄与 = weight / (60 + rank)
```

順位は1から始まります。現行のHTTP/CLIリクエストでは、語彙検索の重みとベクトル検索の重みはいずれも1.0です。両方に出たIDは二つの寄与を加算し、片方だけのIDも残します。最終結果は統合スコアの降順です。同点では候補を統合用配列へ追加した順序を維持します。現行実装は語彙検索の候補を先に追加し、その後にベクトル検索だけに現れた候補を追加します。

レスポンスには次のスコアを別々に返します。

- `lexical_score`: 語彙検索候補にあった場合のBM25Fスコアです。
- `vector_score`: ベクトル候補にあった場合の距離尺度に基づくスコアです。
- `fused_score`: RRFで使う最終スコアです。

語彙検索方式またはベクトル検索方式でも共通の統合処理を通るため、最終順位用の`fused_score`と元スコアを区別してください。

## `scope`

### `documents`

文書IDを結果の単位にします。語彙検索では文書ポスティングを検索します。ベクトルは本文断片のベクトルを検索した後、同じ親文書IDの候補をまとめ、より高い候補を残します。結果は文書ID、題名、URL、抜粋、各スコアを返します。保存済みの`metadata`は現行の検索レスポンスには含まれません。

通常の検索結果画面や、文書へのリンクを1件ずつ表示したい場合に使用します。一つの文書の複数箇所が一致しても、同じ文書が
何度も並ぶことはありません。

### `passages`

本文断片IDを結果の単位にします。語彙検索は本文断片フィールドを、ベクトル検索は本文断片のベクトルを検索します。親文書が新しい登録・更新操作または削除操作で不可視なら、その本文断片も除外します。

RAG向けの`/v2/retrieve`は常に本文断片を検索します。

検索語と一致した本文の場所をそのまま確認したい場合や、LLMへ渡す参照資料を選ぶ場合に使用します。同じ文書から複数の
本文断片が結果へ入ることがあります。

## メタデータによる絞り込み

`filter`は`metadata.filterable_fields`へ事前登録したフィールドだけを参照できます。未登録フィールドを含む`filter`はリクエスト全体を拒否します。深さは最大32、ノード数は最大1024です。

各フィルターノードは、演算子名を一つだけ持つJSONオブジェクトです。

### 等値

```json
{
  "eq": {
    "field": "language",
    "value": "ja"
  }
}
```

null、真偽値、数値、文字列を比較できます。型をまたいだ比較は一致しません。数値は保存値をdoubleへ変換して比較します。

### 集合

```json
{
  "in": {
    "field": "category",
    "values": ["manual", "reference"]
  }
}
```

`values`は空でないスカラー値の配列です。メタデータ側が配列だった場合は、保存されたどれか一要素が一致すれば成立します。

### 範囲

```json
{
  "range": {
    "field": "year",
    "gte": 2020,
    "lt": 2030
  }
}
```

`gt`、`gte`、`lt`、`lte`を一つ以上指定します。数値フィールドだけが対象で、指定した境界をすべて満たす必要があります。

### 存在確認

```json
{
  "exists": {
    "field": "author.country"
  }
}
```

`null`値も項目として存在するため、`exists`は`true`になります。パス自体がない文書だけが`false`です。

### 論理演算

```json
{
  "and": [
    {"eq":{"field":"language","value":"ja"}},
    {"not":{"range":{"field":"year","lt":2020}}}
  ]
}
```

`and`と`or`は空でない条件配列、`not`は条件を一つ取ります。

`filter`は親文書のメタデータへ評価します。`scope = "passages"`でも、本文断片自身に独立したメタデータはありません。

## カーソルによる続きの取得

`POST /v2/search`は、まだ結果がある場合に`next_cursor`を返します。カーソルは次の情報を含む、内部構造を公開しない文字列です。

- カーソル形式の版
- 検索に使用したスナップショットの世代
- 続きの開始位置
- 検索条件のSHA-256から求めたダイジェスト

検索文、ベクトル、検索方式、`scope`、`filter`、`operator`、フレーズ、`limit`のいずれかを変えてカーソルを再利用できません。世代が変わった場合も拒否します。開始位置の上限は10000です。

カーソルは改ざん防止の秘密鍵付き署名ではありません。検索条件と世代の取り違えを検出するダイジェストです。クライアントは内容を組み立てず、応答の値をそのまま次の要求へ渡してください。

## RAG向け`retrieve`

`POST /v2/retrieve`は本文断片検索結果から、LLMへ渡せる`context`と引用情報を組み立てます。回答生成は行いません。

| キー | データ型 | 入力可能値 | デフォルト値 | 必須 | 説明 |
|---|---|---|---|---|---|
| `query` | 文字列 | 空でないUTF-8文字列。正規化後に検索語が残る内容 | なし | `mode`が`lexical`または`hybrid`の場合は必須 | 語彙検索に使う文字列です。リクエスト本文全体には1048576バイトの上限があります。 |
| `vector` | 浮動小数点数の配列 | 要素数は索引の`dimensions`と同数。各要素はfloat32で表現できる有限値 | なし | `mode`が`vector`または`hybrid`の場合は必須 | ベクトル検索に使います。 |
| `mode` | 文字列 | `lexical`、`vector`、`hybrid` | `hybrid` | 任意 | 使用する検索方式を指定します。 |
| `filter` | フィルターオブジェクト | 深さ32以下、ノード数1024以下 | なし | 任意 | `metadata.filterable_fields`へ登録した値で候補を絞り込みます。 |
| `operator` | 文字列 | `or`、`and` | `or` | 任意 | `or`は一部の検索語への一致を許し、`and`はすべての検索語への一致を必要とします。 |
| `phrase` | 真偽値 | `true`、`false` | `false` | 任意 | `true`では検索語が同じ順序で連続する候補だけを残します。 |
| `limit` | 整数 | 1〜100 | `20` | 任意 | 採用する本文断片数の上限です。 |
| `max_passages_per_document` | 整数 | 1〜`limit` | `3` | 任意 | 同じ文書から採用する本文断片数を制限します。 |
| `max_context_bytes` | 整数 | 1〜1048576 | `16384` | 任意 | `context`へ連結する本文のUTF-8バイト数上限です。 |

順位順に本文断片を調べ、同じ本文断片の重複と1文書あたりの上限を除きます。`context`では、選んだ本文の間に改行を二つ入れます。次の本文全体が残りバイト数へ入らない場合は、途中で切らずにその本文断片を飛ばします。

各引用情報には、本文断片ID、文書ID、タイトル、URL、本文、元本文の文字位置、コンテキスト内のバイト開始・終了位置、三種類のスコアを返します。セグメント通し番号、オブジェクト通し番号、ID、親文書IDが一致しない場合はスナップショット競合として処理を止めます。

## 抜粋

検索応答の`snippet`は、検索語の周辺を結果一覧へ表示するための短い抜粋です。文書検索では本文から、本文断片検索では
その本文断片から取り出します。長い本文は検索語の前後を含むおよそ180文字分へ短くします。UTF-8の途中や、結合文字・
絵文字の並びを構成する途中では切らないように処理します。これは表示用の概要であり、元本文の完全な引用ではありません。
LLMへ渡す本文と出典情報が必要な場合は`POST /v2/retrieve`を使用してください。

## 検索結果を解釈するときの注意

- `limit`は該当総数を保証する値ではなく、返す上限です。
- USearchによるベクトル検索は候補を近似的に選びます。公開スコアは元ベクトルから再計算しますが、候補に入らなかった近いベクトルは返りません。
- 複合検索の`fused_score`は順位統合用で、語彙検索またはベクトルの意味的な絶対値ではありません。
- `filterable_fields`を後から追加しても既存索引には値がありません。元文書から再索引します。
- 検索ベクトルのモデルIDをリクエストに含めるフィールドはありません。呼び出し側が索引のモデルIDと生成設定を管理します。
