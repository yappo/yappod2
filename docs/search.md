# 検索の仕組みと指定方法

Yappod2は、語句による語彙検索、事前計算したベクトルによる類似検索、両方の順位を統合する複合検索を提供します。検索結果は文書単位またはパッセージ単位で返せます。外部APIのリクエストとレスポンスの全項目は[`yappod_front` API](yappod-front-api.md)を参照してください。

## 共通する検索処理

すべての検索は、一つのマニフェスト世代から開いたスナップショットに対して実行します。検索中に更新が公開されても、処理途中で新旧セグメントを混ぜません。

複数セグメントに同じ文書IDがある場合は新しいセグメントを優先します。新しい削除標識がある文書は候補から除きます。パッセージの候補も親文書が現在有効かを確認します。

HTTP APIの既定値は次のとおりです。

| 項目 | 既定値 | 範囲 |
|---|---|---|
| `mode` | `hybrid` | `lexical`、`vector`、`hybrid` |
| `scope` | `documents` | `documents`、`passages`。`retrieve`では`passages`固定です。 |
| `operator` | `or` | `or`、`and` |
| `phrase` | `false` | 真偽値 |
| `limit` | `20` | 1〜100 |
| 内部候補数 | 最低100 | 続きを取得するときは、必要な開始位置まで増やします。 |

CLI `search`だけは`limit = 10`を既定としてリクエストへ入れます。

## 語彙検索

検索文を索引と同じトークナイザーで正規化・分割し、題名、本文、パッセージの転置索引を検索します。

### BM25F

既定値は次のとおりです。

| 係数 | 値 |
|---|---:|
| `k1` | 1.2 |
| `b` | 0.75 |
| 題名の重み | 2.0 |
| 本文の重み | 1.0 |
| パッセージの重み | 1.0 |

各フィールドの語句頻度を、そのフィールドの平均長と対象フィールドの長さで正規化し、重みを掛けて合算します。逆文書頻度はセグメント内の対象数と文書頻度から計算します。題名に同じ語がある文書は、同条件なら本文だけにある文書より高くなります。

`terms.yap2`から検索語を二分探索し、`postings.yap2`の128件単位のブロックを読みます。ブロックに保存した最大語句頻度と最小フィールド長から上限スコアを求め、現在の上位`k`件へ届かないブロックをBlock-Max WANDで飛ばします。最終スコアは転置リストの実値から計算します。

### `operator`

- `or`は、一つ以上の検索トークンが現れるオブジェクトを候補にします。
- `and`は、すべての検索トークンが同じオブジェクトに現れることを要求します。

正規化後に同じトークンが複数回現れる検索文では、語彙状態を重複して作らず処理します。

### `phrase`

`phrase = true`では、すべての検索トークンが同じフィールド内で検索文の順に連続して現れることを位置情報から確認します。フレーズを有効にすると、実質的に全トークン一致も必要です。題名から本文へまたがる一致や、別のパッセージへまたがる一致はフレーズになりません。

## ベクトル検索

ベクトル検索は検索文から埋め込みを生成しません。呼び出し側が索引と同じモデル、前処理、`dimensions`で検索ベクトルを作ります。search-webでは`[embedding]`が設定されている場合にBFFが担当します。

リクエストのベクトルは、要素数が索引`dimensions`と同じで、全要素がfloat32へ変換できる有限値でなければなりません。索引が`vector.enabled = false`の場合、ベクトル検索または複合検索のリクエストは不正です。

### 距離尺度とscore

| metric | score | 並び順 |
|---|---|---|
| `cosine` | 二つのベクトルのcosine類似度 | 大きい順です。0 normを含む場合は0です。 |
| `dot` | 内積 | 大きい順です。 |
| `l2` | 二乗Euclidean distanceの負値 | 0に近い大きな値ほど近い結果です。 |

`vectors.usearch`があるセグメントではUSearchのHNSWから候補通し番号を得ます。結果として公開するスコアは、`vectors.yap2`の元のfloat32ベクトルからYappod2が再計算します。ANNがないセグメントを読み取り処理へ渡した場合は全件走査へ切り替える実装ですが、通常の索引作成はベクトルとANNを一緒に作成し、マニフェスト検証で両者の整合を確認します。

## 複合検索

語彙検索とベクトルの生スコアは尺度が異なるため、直接加算しません。それぞれの候補をスコア順に並べ、Reciprocal Rank Fusionで統合します。

```text
寄与 = weight / (60 + rank)
```

順位は1から始まります。現行のHTTP/CLIリクエストでは、語彙検索の重みとベクトル検索の重みはいずれも1.0です。両方に出たIDは二つの寄与を加算し、片方だけのIDも残します。最終結果は統合スコアの降順です。同点では候補を統合用配列へ追加した順序を維持します。現行実装は語彙検索の候補を先に追加し、その後にベクトル検索だけに現れた候補を追加します。

レスポンスには次のスコアを別々に返します。

- `lexical_score`: 語彙検索候補にあった場合のBM25Fスコアです。
- `vector_score`: ベクトル候補にあった場合の距離尺度に基づくスコアです。
- `fused_score`: RRFで使う最終スコアです。

語彙検索方式またはベクトル検索方式でも共通の統合処理を通るため、最終順位用の`fused_score`と元スコアを区別してください。

## `scope`

### `documents`

文書IDを結果の単位にします。語彙検索では文書ポスティングを検索します。ベクトルはパッセージのベクトルを検索した後、同じ親文書IDの候補をまとめ、より高い候補を残します。結果は文書ID、題名、URL、抜粋、各スコアを返します。保存済みの`metadata`は現行の検索レスポンスには含まれません。

### `passages`

パッセージIDを結果の単位にします。語彙検索はパッセージフィールドを、ベクトル検索はパッセージのベクトルを検索します。親文書が新しい登録・更新操作または削除操作で不可視なら、そのパッセージも除外します。

RAG向けの`/v2/retrieve`は常にパッセージを検索します。

## メタデータによる絞り込み

`filter`は`metadata.filterable_fields`へ事前登録したフィールドだけを参照できます。未登録フィールドを含む`filter`はリクエスト全体を拒否します。深さは最大32、ノード数は最大1024です。

各フィルターノードは、演算子名を一つだけ持つJSONオブジェクトです。

### 等値

```json
{"eq":{"field":"language","value":"ja"}}
```

null、真偽値、数値、文字列を比較できます。型をまたいだ比較は一致しません。数値は保存値をdoubleへ変換して比較します。

### 集合

```json
{"in":{"field":"category","values":["manual","reference"]}}
```

`values`は空でないスカラー値の配列です。メタデータ側が配列だった場合は、保存されたどれか一要素が一致すれば成立します。

### 範囲

```json
{"range":{"field":"year","gte":2020,"lt":2030}}
```

`gt`、`gte`、`lt`、`lte`を一つ以上指定します。数値フィールドだけが対象で、指定した境界をすべて満たす必要があります。

### 存在確認

```json
{"exists":{"field":"author.country"}}
```

`null`値も項目として存在するため、`exists`は`true`になります。パス自体がない文書だけが`false`です。

### 論理演算

```json
{
  "and": [
    {"eq":{"field":"language","value":"ja"}},
    {"not":{"range":{"field":"year","lt":2020}}}
  ]
}
```

`and`と`or`は空でない条件配列、`not`は条件を一つ取ります。

`filter`は親文書のメタデータへ評価します。`scope = "passages"`でも、パッセージ自身に独立したメタデータはありません。

## カーソルによる続きの取得

`POST /v2/search`は、まだ結果がある場合に`next_cursor`を返します。カーソルは次の情報を含む、内部構造を公開しない文字列です。

- カーソル形式の版
- 検索に使用したスナップショットの世代
- 続きの開始位置
- 検索条件のSHA-256から求めたダイジェスト

検索文、ベクトル、検索方式、`scope`、`filter`、`operator`、フレーズ、`limit`のいずれかを変えてカーソルを再利用できません。世代が変わった場合も拒否します。開始位置の上限は10000です。

カーソルは改ざん防止の秘密鍵付き署名ではありません。検索条件と世代の取り違えを検出するダイジェストです。クライアントは内容を組み立てず、応答の値をそのまま次の要求へ渡してください。

## RAG向け`retrieve`

`POST /v2/retrieve`はパッセージ検索結果から、LLMへ渡せる`context`と引用情報を組み立てます。回答生成は行いません。

| 項目 | 既定値 | 範囲 |
|---|---:|---|
| `limit` / 最大パッセージ数 | 20 | 1〜100 |
| `max_passages_per_document` | 3 | 1〜`limit` |
| `max_context_bytes` | 16384 | 1〜1048576 |

順位順にパッセージを調べ、同じパッセージの重複と1文書あたりの上限を除きます。`context`では、選んだ本文の間に改行を二つ入れます。次の本文全体が残りバイト数へ入らない場合は、途中で切らずにそのパッセージを飛ばします。

各引用情報には、パッセージID、文書ID、タイトル、URL、本文、元本文の文字位置、コンテキスト内のバイト開始・終了位置、三種類のスコアを返します。セグメント通し番号、オブジェクト通し番号、ID、親文書IDが一致しない場合はスナップショット競合として処理を止めます。

## 抜粋

検索応答の`snippet`は、対象本文から検索文付近を最大180書記素を目安に取り出します。UTF-8や書記素の途中で切りません。`snippet`は表示用です。RAGへ渡す引用本文が必要な場合は`retrieve`を使用してください。

## 検索結果を解釈するときの注意

- `limit`は該当総数を保証する値ではなく、返す上限です。
- ANNは近似候補取得です。公開スコアは再計算しますが、候補に入らなかった近傍は返りません。
- 複合検索の`fused_score`は順位統合用で、語彙検索またはベクトルの意味的な絶対値ではありません。
- `filterable_fields`を後から追加しても既存索引には値がありません。元文書から再索引します。
- 検索ベクトルのモデルIDをリクエストに含めるフィールドはありません。呼び出し側が索引のモデルIDと生成設定を管理します。
