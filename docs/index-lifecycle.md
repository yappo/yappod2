# 索引の更新と保守

## 更新とgeneration

`build`は最初の公開generationを作成します。`update`は最大100操作を検証し、作成後に変更しないセグメントを
作ってから、次のgenerationを不可分に公開します。検索中のスナップショットは解放されるまで同じgenerationを読み続けます。

## セグメントの分割

分割計画は文書とそのパッセージを分離せず、文書境界でセグメントを分けます。文書、語彙索引、メタデータ、ベクトルの
各ペイロードが256 MiB以内になるよう見積もり、索引作成、更新、コンパクションで同じ規則を使います。単独文書が上限へ
収まらない場合は、その文書を受理せず明示的なエラーを返します。

## compactionの実行

コンパクションは現在有効な文書とパッセージだけを新しいセグメントへ書きます。新しいmanifestの公開後、参照されなくなった
セグメントを回収します。書き込み用ロックにより更新との同時実行を防ぎます。失敗時は公開済みmanifestを変更しません。

## 起動時の検証

起動時は設定の指紋値、manifest、コンポーネントの大きさとヘッダー、必要な内部構造を検証してスナップショットを開きます。
リクエストごとに全ファイルのSHA-256を再計算しません。これにより検索経路で大きな索引を毎回走査することを避けます。

## 全検証

```sh
./build/yappo_makeindex verify --config application.toml
```

手動検証ではmanifestから参照される全コンポーネントのチェックサムと内部構造を確認します。定期監視、配布前、
ストレージ障害後に実行してください。成功時は終了状態0です。

## 障害復旧

公開前に作成された一時ファイルや、参照されないセグメントは公開済みmanifestから参照されません。復旧ではまずmanifestと
ログを保存し、`verify`で公開スナップショットを確認します。manifestが参照しているファイルを手作業で削除しないでください。
