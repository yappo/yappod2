# 索引の更新と保守

この文書では、索引の作成、更新、読み替え、コンパクション、検証、障害復旧を説明します。ディスク上の各ファイルとバイナリ形式は[索引ファイル形式](index-format.md)、コマンドの全引数は[コマンドリファレンス](command-reference.md)を参照してください。

## 世代とスナップショット

`manifest.json`の`generation`は、公開済み索引の世代番号です。`build`はまず空の世代1を一時索引内に作り、最初の入力バッチを世代2として公開します。入力が10000件を超える場合は内部バッチごとに世代が進みます。`update`と`yappo_compact`も成功するたびに1増やします。検索処理は、特定の世代に対応するスナップショットを取得してから処理を始めます。処理中に新しい世代が公開されても、そのリクエストは取得済みのスナップショットを最後まで使用します。

`yappod_core`は起動時にスナップショットを開き、その後は1秒間隔で`manifest.json`を確認します。新しい世代を正常に検証できた場合だけ現在のスナップショットと交換します。読み込みに失敗した候補を検索へ使うことはありません。

## 最初の索引を作る

```sh
./build/yappo_makeindex build \
  --config examples/config.lexical.toml \
  --input documents.ndjson
```

`build`は次の順序で処理します。

1. アプリケーション用TOMLとNDJSONを検証します。
2. 1万件ずつ内部処理し、文書境界を保つセグメント分割を計画します。この1万件はメモリー使用量を抑える内部単位であり、入力全体の件数上限ではありません。
3. 新しい索引ディレクトリへ`config.toml`、各セグメント、`manifest.json`を書きます。
4. 各コンポーネントの大きさ、チェックサム、内部構造を検証してから公開を完了します。

出力先は存在していてはいけません。空のディレクトリであっても`build`は上書きしません。運用中の索引ディレクトリへ`build`を重ねて実行しないでください。構成を変更する場合は別のディレクトリへ作り、検証後にアプリケーション用TOMLの`[index].directory`を切り替えます。

## 文書を追加、置換、削除する

```sh
./build/yappo_makeindex update \
  --config examples/config.lexical.toml \
  --input operations.ndjson
```

NDJSONの1行が1件の更新操作です。`update`は1回につき1〜100件を受け付け、入力全体を1つの世代として公開します。

- `upsert`は新しい文書を追加します。同じIDがすでに存在する場合は新しい文書を優先し、古い文書を検索結果から隠します。
- `delete`は指定IDを削除標識（tombstone）へ記録します。存在しないIDを指定しても、削除操作として受理します。
- 同じ入力内に同一IDが複数回現れる場合は不正です。
- 1行でも不正なら、その入力による新しい世代は公開しません。

更新処理は`writer.lock`を排他ロックします。`update`と`yappo_compact`を同じ索引に対して同時実行した場合、片方はロックを取得するまで待ちます。ロックファイルそのものが存在することは、別の処理が動いている証拠ではありません。プロセスが保持している`flock`が排他制御を行います。

## 更新を公開する仕組み

更新は、既存のセグメントを変更せずに新しいセグメントを作成します。書き込みと検証が終わると、更新開始時の世代が変わっていないことを確認して`manifest.json`を置き換えます。この比較と置換により、別の書き込みが先に公開された場合は`generation_conflict`として失敗します。

公開前に失敗した新しいセグメントはマニフェストから参照されません。公開済みマニフェストが指すセグメントを途中まで書き換える方式ではないため、検索側から未完成の更新は見えません。

## セグメントを分ける条件

文書とその文書から作ったパッセージは同じ分割単位です。文書、語彙、メタデータ、ベクトルの各コンポーネントが、それぞれ256 MiBのペイロード上限へ収まるように文書境界で分割します。`build`、`update`、`yappo_compact`は同じ分割計画を使用します。

単独の文書だけでいずれかのコンポーネントが上限を超える場合、文書を途中で別セグメントへ分けず、その文書ID、対象コンポーネント、必要なバイト数、上限を含むエラーを返します。本文を小さくするか、アプリケーション側で文書を分割してください。

## コンパクション

更新を重ねると、古い版の文書と削除標識を含むセグメントが残ります。コンパクションは、現在見えている文書とパッセージだけを読み出し、新しいセグメント群として書き直します。

```sh
./build/yappo_compact --config examples/config.lexical.toml
```

処理は次の順序で進みます。

1. `writer.lock`を取得します。
2. 現在の設定、マニフェスト、全コンポーネントを検証します。
3. マニフェストから参照されていない古いセグメントディレクトリを回収します。
4. 現在見えている文書、パッセージ、ベクトルを新しいセグメントへ書きます。
5. 新しいセグメントだけを指すマニフェストを次の世代として公開します。
6. 旧マニフェストだけが参照していたセグメントを回収します。

成功時は世代、文書数、パッセージ数、削除したセグメント数、新しいセグメントIDをJSONで標準出力へ返します。コンパクション中も、デーモンは公開済みの旧世代を検索できます。新しいマニフェストを公開した後は、coreの再読み込みを待って新しい世代へ移ります。

## `compaction.state`の読み方

コンパクションは索引直下の`compaction.state`へ、次の1行をタブ区切りで書きます。

```text
YAP2-COMPACTION\trunning\t12345\t8\t1784512345
```

各列は形式名、状態、プロセスID、世代、更新時刻のUnix秒です。状態は`running`、`succeeded`、`failed`です。`running`のプロセスが存在しなければ、ヘルスチェックとメトリクスでは`interrupted`として扱います。ファイルがなければ`idle`、内容を読めなければ`unknown`です。このファイルは監視情報であり、マニフェストの代わりにはなりません。

## 起動時の検証と全検証

デーモン起動時と世代の再読み込み時は、設定とマニフェストの整合、必要なコンポーネント、ヘッダー、件数、参照範囲など、検索に必要な構造を確認します。リクエストごとに全ファイルのSHA-256を再計算することはありません。

保存や配布の前、ストレージ障害の後、手作業でファイルを移動した後は、全検証を実行します。

```sh
./build/yappo_makeindex verify --config examples/config.lexical.toml
```

`verify`は`manifest.json`から参照される各ファイルの大きさとSHA-256を照合し、共通ヘッダー、レコード数、オフセット、転置リスト、出現位置、メタデータ、ベクトル、近似最近傍探索用データの対応まで読み取ります。成功時は標準出力へ世代とセグメント数を表示し、終了状態0を返します。

## 索引構成を変更する

`[vector]`、トークナイザー、文書分割、絞り込み対象フィールドは索引の保存形式と互換性に関わります。アプリケーション用TOMLだけを変えて既存索引を開くことはできません。特に`[vector].enabled = true`へ変えても、既存の語彙索引へベクトルは追加されません。

安全な切り替えは次の手順です。

1. 元の正式入力または再生成可能な入力から、新しい設定に対応するNDJSONを作ります。
2. 既存とは別の索引ディレクトリへ`build`します。
3. 新しい索引の`config.toml`と`manifest.json`を確認し、`verify`を実行します。
4. 一時的なデーモンまたは`search`で代表的な検索を確認します。
5. アプリケーション用TOMLの`[index].directory`を新しい絶対パスへ切り替えます。
6. coreとfrontを停止し、新しい設定で起動します。

古い索引は、切り替え後の確認と必要な保存期間が終わるまで残してください。

## バックアップと復元

整合したバックアップには、同じ時点の`config.toml`、`manifest.json`、マニフェストが参照するすべての`segments/<id>/`が必要です。ファイルを個別にコピーしている途中に更新が公開されると世代が混ざる可能性があります。ストレージのスナップショット機能を使うか、更新とコンパクションを止めた状態でディレクトリ全体を取得してください。

復元後は、`[index].directory`が復元先を指すアプリケーション用TOMLを用意し、デーモンを起動する前に`yappo_makeindex verify --config <設定ファイル>`を実行します。アプリケーション用TOMLだけを保存しても索引は復元できません。逆に`writer.lock`、`manifest.json.lock`、`compaction.state`は制御・監視用であり、検索内容を構成するデータではありません。

## 障害時の確認順

1. 索引ディレクトリ全体とデーモンのログを変更せず保存します。
2. `manifest.json`の世代と参照セグメントを確認します。
3. `compaction.state`が`running`のままなら、記録されたプロセスIDが存在するか確認します。
4. `verify`を実行し、最初に失敗するファイルと理由を記録します。
5. 公開前に残ったマニフェスト非参照セグメントは、次回のコンパクションが回収できます。原因調査前に手作業で削除しないでください。
6. マニフェストが参照するファイルが欠けるか壊れている場合は、検証済みバックアップから索引全体を復元するか、正式入力から別ディレクトリへ再構築します。

マニフェストやコンポーネントを手作業で修正すると、チェックサムだけでなくレコード間の対応を壊します。修復用の正式なコマンドは実装されていないため、手編集による復旧は行わないでください。
