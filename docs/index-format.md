# 索引ファイル形式

この文書は、Yappod2 v2索引のディレクトリ構造、マニフェスト、各コンポーネントのバイナリ形式、読込時の検証規則を説明します。バックアップ、破損調査、互換性確認、別言語での検査ツール作成に必要な情報を対象にしています。

索引ファイルを直接編集する運用はサポートしていません。更新には`yappo_makeindex update`、統合には`yappo_compact`を使用してください。

## セグメントとは何か

Yappod2は、索引全体を一つの巨大なファイルへ保存しません。文書をまとめて登録するたびに、その文書を検索するために
必要なファイル一式を新しいディレクトリへ保存します。この一式を「セグメント」と呼びます。

たとえば最初に1000文書を登録すると、文書本文、検索語、検索語の出現位置、メタデータ、ベクトルなどを持つセグメントが
作られます。その後で10文書を更新すると、最初のセグメントは書き換えず、更新した10文書を持つ新しいセグメントを追加します。
同じ文書IDが複数のセグメントにある場合は、新しいセグメントの内容を使用します。削除した文書IDも新しいセグメントへ
記録し、古いセグメントに残る文書を検索結果から除外します。

`manifest.json`は、現在の検索に使用するセグメントを古い順に並べた一覧です。検索開始時にこの一覧を読み、その時点の
セグメントだけを組み合わせて検索します。コンパクションは、現在有効な文書を新しい少数のセグメントへまとめ直し、
古いセグメントを一覧から外す処理です。

## ディレクトリ構成

```text
<index-directory>/
├── config.toml
├── manifest.json
├── manifest.json.lock
├── writer.lock
├── compaction.state
└── segments/
    ├── seg-<generation>-<random>/
    │   ├── documents.yap2
    │   ├── terms.yap2
    │   ├── postings.yap2
    │   ├── positions.yap2
    │   ├── metadata.yap2
    │   ├── vectors.yap2
    │   ├── vectors.usearch
    │   └── tombstones.yap2
    └── compact-<generation>-<random>/
        └── ...
```

常に必要なのは`config.toml`、`manifest.json`、`segments/`です。ロックファイルとコンパクション状態ファイルは、更新処理を実行した後に存在する場合があります。セグメント内で必須・任意のファイルは次のとおりです。

各セグメントでは、同じ文書と本文断片を複数のファイルから参照します。`documents.yap2`が本文とIDを保持し、
`terms.yap2`、`postings.yap2`、`positions.yap2`が「どの検索語が、どの文書の、どこに現れたか」を保持します。
`metadata.yap2`は絞り込み、`vectors.yap2`と`vectors.usearch`はベクトル検索、`tombstones.yap2`は削除に使用します。
`manifest.json`では、これら個々のファイルを`components`配列へ記録するため、この文書でも個々の索引ファイルを
「コンポーネント」と表記します。

| ファイル | マニフェストの`file_type` | 存在条件 | 役割 |
|---|---:|---|---|
| `documents.yap2` | 4 | すべてのセグメントで必須 | 文書と生成済み本文断片の本文・属性を保存します。削除だけのセグメントでは件数0のファイルになります。 |
| `terms.yap2` | 1 | 登録・更新操作を含むセグメントで必須 | 語彙をUTF-8バイト順に並べ、ポスティングと位置情報の範囲を指します。 |
| `postings.yap2` | 2 | 登録・更新操作を含むセグメントで必須 | 語彙ごとの文書・本文断片出現、頻度、フィールド長、読み飛ばし用ブロック情報を保存します。 |
| `positions.yap2` | 3 | 登録・更新操作を含むセグメントで必須 | 語彙が出現するフィールドと位置を保存します。フレーズ検索に使います。 |
| `metadata.yap2` | 5 | 登録・更新操作を含むセグメントで必須 | 絞り込み対象のスカラー値を保存します。`filterable_fields`が空なら、フィールド数と値件数が0のファイルになります。 |
| `vectors.yap2` | 6 | ベクトルが有効で、本文断片があるセグメント | 本文断片IDとfloat32ベクトルを保存します。 |
| `vectors.usearch` | 8 | `vectors.yap2`があるセグメント | 近いベクトルの候補を高速に探すため、USearchが作成する検索用ファイルです。Yappod2の共通ヘッダーは持ちません。 |
| `tombstones.yap2` | 7 | 削除操作を含むセグメント | 削除する文書IDを保存します。登録・更新操作と削除操作を含む同一セグメントではほかのファイルと共存します。 |

一つのセグメントがマニフェストへ持てるコンポーネントは最大8個です。索引全体のセグメント数上限は100000です。

## `config.toml`

索引を作成した時点で適用された構造設定だけを保存します。アプリケーション用TOMLから`[index]`、`[daemon]`、`[embedding]`、`[web]`、`[llm]`などを除き、デフォルトを展開して次の形で書きます。

```toml
format_version = 2

[tokenizer]
id = "unicode_nfkc_casefold_v2"

[chunking]
max_chars = 1200
overlap_chars = 200

[vector]
enabled = false

[metadata]
filterable_fields = ["category", "language"]
```

ベクトルが有効なら`model_id`、`dimensions`、`metric`も保存します。`filterable_fields`は辞書順に並べます。次の正規化テキストをSHA-256へ入力し、マニフェストの`config_fingerprint`と照合します。

```text
format_version=<整数>
tokenizer.id=<文字列>
chunking.max_chars=<整数>
chunking.overlap_chars=<整数>
vector.model_id=<文字列>
vector.dimensions=<整数>
vector.metric=disabled|cosine|dot|l2
metadata.filterable_fields[0]=<文字列>
...
```

このファイルは索引形式の一部です。アプリケーション用TOMLの設定変更を反映する目的で書き換えてはいけません。

## `manifest.json`

マニフェストは、現在公開されているスナップショットを決める唯一の一覧です。ファイルサイズは最大16 MiBです。JSONオブジェクトは未知のキーを許さず、次の構造を取ります。

```json
{
  "format_version": 2,
  "generation": 3,
  "config_fingerprint": "64文字の小文字16進SHA-256",
  "segments": [
    {
      "id": "seg-00000000000000000003-AbCd12",
      "documents": 2,
      "passages": 4,
      "tombstones": 1,
      "components": [
        {
          "name": "documents.yap2",
          "file_type": 4,
          "records": 6,
          "file_bytes": 12345,
          "sha256": "64文字の小文字16進SHA-256"
        }
      ]
    }
  ]
}
```

### 最上位フィールド

| フィールド | 型と条件 | 説明 |
|---|---|---|
| `format_version` | 符号なし整数、`2`固定 | マニフェスト形式の版です。 |
| `generation` | 符号なし64ビット整数相当、1以上 | スナップショットの世代です。更新公開ごとに1増えます。 |
| `config_fingerprint` | 64文字の小文字16進数 | `config.toml`から求めるSHA-256です。全0は禁止されます。 |
| `segments` | 配列、最大100000件 | 古いものから新しいものの順です。IDは重複できません。 |

### セグメント記述子

| フィールド | 型と条件 | 説明 |
|---|---|---|
| `id` | 空でない文字列、最大255バイト | `segments/`直下のディレクトリ名です。英数字、`-`、`_`、`.`だけを許可します。通常はYappod2が`seg-...`または`compact-...`の形で生成します。 |
| `documents` | 符号なし整数 | セグメントに保存した文書数です。 |
| `passages` | 符号なし整数 | セグメントに保存した本文断片数です。 |
| `tombstones` | 符号なし整数 | セグメントに保存した削除ID数です。 |
| `components` | 1〜8件の配列 | ファイル記述子です。同じ名前または同じ`file_type`を重複できません。 |

### コンポーネント記述子

| フィールド | 型と条件 | 説明 |
|---|---|---|
| `name` | 1〜63バイト、`/`と`\\`は禁止 | セグメントディレクトリ内のファイル名です。 |
| `file_type` | 1〜8 | 上表の種類番号です。 |
| `records` | 符号なし整数 | コンポーネント固有のレコード数です。文書ファイルなら文書と本文断片の合計、メタデータファイルなら値の件数です。 |
| `file_bytes` | 符号なし整数、1以上 | 共通ヘッダーを含むファイル全体のサイズです。`vectors.usearch`ではUSearchが出力したファイル全体のサイズです。 |
| `sha256` | 64文字の小文字16進数 | ファイル全体のSHA-256です。 |

同じ文書IDが複数セグメントに現れる場合は、配列の後ろにある新しいセグメントを優先します。削除記録も同じ順序で古い文書を隠します。

マニフェストは`manifest.json.tmp`へ書き、ファイルを`fsync`し、`manifest.json`へ名前変更し、親ディレクトリを`fsync`して公開します。`manifest.json.lock`はこの公開処理の`flock`に使います。

## `.yap2`共通ヘッダー

`vectors.usearch`以外のコンポーネントは32バイトの共通ヘッダーを持ちます。すべての整数と浮動小数点数のビット列はリトルエンディアンです。C構造体のパディングやポインターをファイルへ書きません。

| オフセット | バイト数 | 型 | フィールド | 許容値 |
|---:|---:|---|---|---|
| 0 | 4 | byte[4] | マジック | ASCII `YAP2` |
| 4 | 2 | uint16 | `format_version` | `2` |
| 6 | 2 | uint16 | `header_bytes` | `32` |
| 8 | 4 | uint32 | `file_type` | 1〜7 |
| 12 | 8 | uint64 | `generation` | 1以上。そのコンポーネントを作成した世代です。現在のマニフェストの`generation`以下でなければなりません。同じセグメント内の`.yap2`ファイルでは一致します。 |
| 20 | 8 | uint64 | `payload_bytes` | ファイルサイズから32を引いた値です。最大256 MiBです。 |
| 28 | 4 | uint32 | `payload_crc32c` | ペイロード全体のCRC32Cです。多項式の反転表現は`0x82f63b78`です。 |

マニフェストのSHA-256はヘッダーとペイロードを含むファイル全体を保護し、ヘッダーのCRC32Cはペイロードを独立に検査します。

以下の表で`string`は`uint32 byte_length`に続くUTF-8の生バイト列です。NUL終端は保存しません。文字列内のNULは許可されません。

## `documents.yap2`

このファイルは、検索結果を利用者へ返すために必要な元データを保存します。文書レコードには文書ID、URL、題名、本文、
メタデータ、更新時刻が入ります。本文断片レコードには、分割後の本文、元の文書ID、文書内での順番と位置が入ります。

文書レコードと本文断片レコードには、ファイルへ書いた順番に0から始まる通し番号が付きます。検索語のファイルや
ベクトルのファイルは長いID文字列を繰り返し保存せず、この通し番号で`documents.yap2`のレコードを参照します。
検索で候補が決まると、Yappod2は通し番号からこのファイルのレコードを読み、レスポンスの`id`、`title`、`url`、
`snippet`などを組み立てます。

たとえば、文書Aから本文断片A-0とA-1、文書Bから本文断片B-0を作ったセグメントでは、ファイル内の概念的な並びは
次のようになります。

```text
文書 0: 文書A
文書 1: 文書B
本文断片 0: A-0（親文書は文書A、ordinalは0）
本文断片 1: A-1（親文書は文書A、ordinalは1）
本文断片 2: B-0（親文書は文書B、ordinalは0）
```

実際のバイナリでは、先頭に件数を書き、その後へすべての文書レコード、すべての本文断片レコードの順で格納します。

### ペイロード先頭

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | ペイロードの版です。現在は`1`です。 |
| 2 | uint32 | セグメントIDのバイト数です。 |
| 3 | byte[] | セグメントIDです。 |
| 4 | uint64 | 文書数です。最大1000000です。 |
| 5 | uint64 | 本文断片数です。最大4000000です。 |
| 6 | 文書レコードの配列 | 文書を通し番号順に保存します。 |
| 7 | 本文断片レコードの配列 | 本文断片を通し番号順に保存します。 |

### 文書レコード

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | レコード種別です。文書は`1`です。 |
| 2 | uint32 | この後に続くレコードのペイロードのバイト数です。 |
| 3 | string | `id`。1〜255バイトです。 |
| 4 | string | `url`。0〜8192バイトです。 |
| 5 | string | `title`。0〜255バイトです。 |
| 6 | string | `body`。0〜1048576バイトです。 |
| 7 | string | 正規化済みのメタデータJSONオブジェクトです。0〜1048576バイトです。 |
| 8 | uint64 | `updated_at_unix_ms`のビット列です。読込時に符号付き64ビット整数として扱い、値は0以上でなければなりません。 |

文書IDはセグメント内で重複できません。

### 本文断片レコード

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | レコード種別です。本文断片は`2`です。 |
| 2 | uint32 | この後に続くレコードのペイロードのバイト数です。 |
| 3 | string | 本文断片IDです。1〜255バイトです。 |
| 4 | string | 親文書ID。1〜255バイトで、同じセグメント内の文書を参照します。 |
| 5 | string | 分割後の本文です。1〜1048576バイトです。 |
| 6 | uint32 | 文書内の0始まり`ordinal`です。 |
| 7 | uint32 | 元本文の`start_char`です。 |
| 8 | uint32 | 元本文の`end_char`です。`start_char`以上です。 |

同じ親文書の本文断片は通し番号が0から連続し、IDも重複できません。

## `terms.yap2`

このファイルは、索引に含まれる検索語の一覧です。各検索語について、実際の出現先を保存する`postings.yap2`と、
出現位置を保存する`positions.yap2`のどこを読めばよいかを記録します。検索時は正規化した検索語をこの一覧から二分探索し、
見つかった項目が指す範囲だけを後続ファイルから読みます。

ペイロードは次のヘッダーに、語句の項目を辞書順で続けます。

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | ペイロードの版`1`です。 |
| 2 | uint64 | 語句数です。 |

各語句の項目は次の形式です。

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 + byte[] | 語句のバイト数とUTF-8のトークンです。 |
| 2 | uint64 | 文書頻度です。文書または本文断片の異なるオブジェクト数です。 |
| 3 | uint64 | `postings.yap2`のペイロード先頭からのオフセットです。 |
| 4 | uint64 | この語句が使うポスティングのバイト数です。 |
| 5 | uint64 | `positions.yap2`のペイロード先頭からのオフセットです。 |
| 6 | uint64 | この語句が使う位置情報のバイト数です。 |

語句は空でなく、UTF-8のバイト辞書順で厳密に昇順です。オフセットと長さは、後述の語句ブロック境界へ正確に一致する必要があります。

## `postings.yap2`

このファイルは、検索語ごとに「どの文書または本文断片へ、何回現れたか」を保存する転置索引です。`terms.yap2`の
一つの検索語が、このファイル内の一つの語句ブロックを指します。各レコードは`documents.yap2`の文書または本文断片の
通し番号を持つため、検索結果の候補を元のIDと本文へ戻せます。語句頻度とフィールド長はBM25Fのスコア計算に使います。

### ペイロードヘッダー

| オフセット | 型 | 内容 |
|---:|---|---|
| 0 | uint32 | ペイロードの版`1`です。 |
| 4 | uint32 | ブロックの大きさ`128`です。 |
| 8 | uint64 | 文書数です。 |
| 16 | uint64 | 本文断片数です。 |
| 24 | uint64 | 全語句のポスティング数の合計です。 |
| 32 | uint64 | タイトルフィールドのトークン数合計です。 |
| 40 | uint64 | 本文フィールドのトークン数合計です。 |
| 48 | uint64 | 本文断片フィールドのトークン数合計です。 |

ヘッダーの後に、語句ごとのブロックを語句の通し番号順に保存します。

### 単語ブロックのヘッダー

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint64 | 語句の通し番号です。 |
| 2 | uint64 | 文書頻度です。 |
| 3 | uint32 | 128件単位のブロック数です。 |

### ポスティングレコード

1件は固定48バイトです。

| オフセット | 型 | 内容 |
|---:|---|---|
| 0 | uint32 | オブジェクト種別です。文書は`1`、本文断片は`2`です。 |
| 4 | uint64 | `documents.yap2`内の文書通し番号または本文断片通し番号です。 |
| 12 | uint32[3] | タイトル、本文、本文断片の語句頻度です。 |
| 24 | uint32[3] | タイトル、本文、本文断片のフィールド長です。 |
| 36 | uint64 | この語句の位置情報列内の開始通し番号です。 |
| 44 | uint32 | 位置情報数です。三フィールドの語句頻度合計と一致します。 |

レコードはオブジェクト種別、オブジェクト通し番号の順に厳密に増加します。

### ポスティングブロックの記述子

ポスティングレコード列の後に、128件ごとに固定16バイトの記述子を置きます。

| オフセット | 型 | 内容 |
|---:|---|---|
| 0 | uint32 | 語句内の先頭ポスティング通し番号です。 |
| 4 | uint32 | このブロックのポスティング件数です。1〜128です。 |
| 8 | uint32 | ブロック内の最大語句頻度です。 |
| 12 | uint32 | 語句が出現するフィールドの最小フィールド長です。 |

最大頻度と最小長は、上限スコアを計算して不要なブロックを飛ばすために使います。検証時にはポスティングレコードから再計算して一致を確認します。

## `positions.yap2`

このファイルは、各検索語が題名、本文、本文断片の何番目に現れたかを保存します。`postings.yap2`の各レコードが
開始位置と件数を持ち、このファイル内の対応する範囲を参照します。`phrase = true`の検索では、複数の検索語の位置が
同じフィールド内で連続しているかを確認します。

ペイロードヘッダーは`uint32 version = 1`と`uint64 position_count`の12バイトです。その後、語句ごとに次を続けます。

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint64 | 語句の通し番号です。 |
| 2 | uint64 | この語句の位置レコード数です。 |
| 3 | 位置レコードの配列 | 各レコードは`uint32 field`と`uint32 position`の8バイトです。 |

フィールドはタイトル=`1`、本文=`2`、本文断片=`3`です。同じポスティング・フィールド内の位置はフィールド長未満で厳密に増加します。

## `metadata.yap2`

このファイルは、`metadata.filterable_fields`へ指定した値だけを検索時の絞り込み用に保存します。元のメタデータJSON全体は
`documents.yap2`にあります。このファイルでは、設定に書いたフィールドの順番と文書の通し番号を使い、たとえば
「文書5の`language`は`ja`」という関係を記録します。`POST /v2/search`の`filter`はこのデータを評価して候補を除外します。

### ペイロードヘッダーとフィールド表

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | ペイロードの版`1`です。 |
| 2 | uint32 | 絞り込み可能フィールド数です。 |
| 3 | uint64 | 文書数です。 |
| 4 | uint64 | メタデータ項目数です。 |
| 5 | string[] | `config.toml`の`filterable_fields`と同じ順序・同じ値です。 |

### メタデータ項目

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | フィールドの通し番号です。 |
| 2 | uint64 | 文書の通し番号です。 |
| 3 | uint32 | 値の種別です。null=`1`、真偽値=`2`、数値=`3`、文字列=`4`です。 |
| 4 | uint32 | 値のバイト数です。 |
| 5 | byte[] | 値です。nullは0バイト、boolは`0x00`または`0x01`、numberは正規化したASCII表記、stringはUTF-8です。 |

設定したJSONパスに配列がある場合は、配列中のスカラー値ごとに項目を作ります。オブジェクトや配列要素としての配列は項目にしません。

## `vectors.yap2`

このファイルは、各本文断片のIDと元のfloat32ベクトルを同じ順番で保存します。本文断片が100件ならベクトルも100件です。
`vectors.usearch`で近い候補の通し番号を得た後、このファイルから対応する本文断片IDと元ベクトルを読み、距離または類似度を
再計算します。モデルID、次元数、計算方法も保存し、索引設定と一致しないファイルを開かないようにしています。

### 固定ヘッダー

ペイロード先頭40バイトは次の形式です。

| オフセット | 型 | 内容 |
|---:|---|---|
| 0 | uint32 | ペイロードの版`1`です。 |
| 4 | uint32 | 距離尺度。cosine=`1`、dot=`2`、l2=`3`です。 |
| 8 | uint32 | 次元数です。 |
| 12 | uint32 | モデルIDのバイト数です。 |
| 16 | uint64 | ベクトルレコード数です。本文断片数と一致します。 |
| 24 | uint64 | ID領域の合計バイト数 |
| 32 | uint64 | ペイロード先頭からfloat32領域までのオフセットです。 |

固定ヘッダーの直後にモデルIDのUTF-8バイト列を置き、その後にベクトルレコード、ID領域、0〜3バイトのゼロ埋め、float32領域を続けます。

### ベクトルレコード

1件は固定16バイトです。

| オフセット | 型 | 内容 |
|---:|---|---|
| 0 | uint64 | ID領域内の本文断片IDのオフセットです。 |
| 8 | uint32 | 本文断片IDのバイト数です。 |
| 12 | uint32 | 予約領域。`0`です。 |

float32領域では、レコード通し番号順に`dimensions`個のIEEE 754 binary32をリトルエンディアンで並べます。NaNと正負の無限大は書き込み時に拒否します。モデルID、次元数、距離尺度は索引`config.toml`と一致しなければ開けません。

## `vectors.usearch`

すべてのベクトルを順番に比較せず、近い可能性が高い候補を先に探すためのファイルです。USearch C APIの
`usearch_save`が出力し、Yappod2独自の共通ヘッダーは付けません。マニフェストには`file_type = 8`、ファイル全体の
サイズとSHA-256、ベクトル件数を記録します。

読込時にはUSearchのメタデータから次を照合します。

- 距離尺度が`vectors.yap2`および索引設定と一致すること
- 量子化形式がfloat32であること
- 次元数が一致すること
- 索引の要素数が`vectors.yap2`のレコード数と一致すること

このファイルだけから本文断片IDを復元しません。USearchのキーは`vectors.yap2`内のレコード通し番号であり、検索結果のIDとスコアは`vectors.yap2`を使って検証・復元します。USearch内部のバイト配置はYappod2の互換仕様として再定義していません。

## `tombstones.yap2`

ペイロードは次の形式です。

| 順序 | 型 | 内容 |
|---:|---|---|
| 1 | uint32 | ペイロードの版`1`です。 |
| 2 | uint64 | 文書ID数です。 |
| 3 | string[] | 削除する文書IDです。各1〜255バイトで、同一ファイル内で重複できません。 |

削除記録はファイルを物理削除する指示ではありません。スナップショットを解決するとき、より古いセグメントにある同じIDの文書と本文断片を非表示にします。コンパクションで有効レコードだけを再構築すると、不要になった削除記録は新しい`compact-*`セグメントへ引き継がれません。

## `writer.lock`

`update`と`yappo_compact`が`open(O_CREAT | O_RDWR, 0600)`し、処理中に排他的`flock`を取得します。内容は使用しません。ファイルが残っていることは、ロックが現在保持されていることを意味しません。PIDファイルとして扱わず、`flock`の状態を確認してください。

## `manifest.json.lock`

マニフェストの次世代を公開する処理が排他的`flock`を取得します。内容は使用しません。書き込みロックは更新処理全体を直列化し、マニフェストロックは読み込み、世代比較、名前変更という公開区間を保護します。

## `compaction.state`

タブ区切りの1行です。

```text
YAP2-COMPACTION<TAB>running|succeeded|failed<TAB>PID<TAB>GENERATION<TAB>UPDATED_AT_UNIX<LF>
```

`running`では実行中PIDを、完了後の`succeeded`と`failed`ではPID `0`を保存します。ヘルスチェック読込時に`running`のPIDが存在しなければ`interrupted`として報告します。不正な形式や読込失敗は`unknown`です。ファイルがなければ`idle`です。

## 一時ファイルと未参照セグメント

- コンポーネントは同じパスに`.tmp`を付けたファイルへ書いてから名前変更します。
- 索引作成は最終索引と同じ親に`<index>.tmp.XXXXXX`を作り、全体成功後に索引名へ名前変更します。
- 更新処理は`segments/seg-<generation>-XXXXXX`、コンパクションは`segments/compact-<generation>-XXXXXX`を作ります。
- マニフェスト公開に失敗した新規セグメントは、処理内で回収します。
- クラッシュで未参照セグメントが残った場合は、次回コンパクションの不要データ回収が現在のマニフェストにないディレクトリを回収します。

現在のマニフェストが参照するセグメントを、手作業で削除してはいけません。

## 読込時の検証順序

`yappo_makeindex verify`とsnapshot openは概ね次を確認します。

1. `config.toml`を解析し、値と索引設定の内部条件を検証します。
2. 設定指紋を計算し、マニフェストの`config_fingerprint`と比較します。
3. マニフェストの正確なキー集合、型、上限、IDとコンポーネントの重複を検証します。
4. 各コンポーネントのファイルサイズとSHA-256をマニフェストと比較します。
5. `.yap2`ヘッダーの種類、世代、ペイロード長、CRC32Cを比較します。
6. 各ペイロードの件数、オフセット、長さ、順序、通し番号、参照範囲を検証します。
7. `vectors.yap2`と`vectors.usearch`、メタデータと設定、語彙三ファイル間の整合性を検証します。

一件でも失敗すればスナップショット全体を開きません。壊れたセグメントだけを無視して検索を継続する動作はありません。

## 容量上限とセグメント分割

各`.yap2`コンポーネントのペイロード上限は256 MiBです。これは索引全体や入力NDJSON全体の上限ではありません。セグメント分割処理は文書境界で入力を複数セグメントへ分け、`documents`、`terms`、`postings`、`positions`、`metadata`、`vectors`、`tombstones`のどれも上限を超えないよう見積もります。

一つの文書と、その文書に属するすべての本文断片・語彙・メタデータ・ベクトルだけでいずれかのコンポーネント上限を超える場合は分割できず、文書ID、必要バイト数、対象コンポーネントを報告して失敗します。

## バックアップと復元

整合したバックアップには、同じ時点の`config.toml`、`manifest.json`、マニフェストから参照される全セグメントが必要です。単一ファイルだけをコピーしても復元できません。安全な方法は更新を止めた状態で索引ディレクトリ全体をコピーすることです。更新中にオンラインコピーする仕組みやスナップショットのエクスポートコマンドは、現在の実装にはありません。
