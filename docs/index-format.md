# 索引形式

## ディレクトリ構成

```text
<index-directory>/
├── config.toml
├── manifest.json
└── segments/
    └── <segment-id>/
        ├── documents.yap2
        ├── terms.yap2
        ├── postings.yap2
        ├── positions.yap2
        ├── metadata.yap2
        ├── vectors.yap2
        ├── vectors.usearch
        └── tombstones.yap2
```

セグメントの内容と設定により、メタデータ、ベクトル、ANN、削除記録のコンポーネントは存在しない場合があります。

`config.toml`には索引構造へ影響するトークン分割、チャンク分割、ベクトル、メタデータの設定だけを保存します。
デーモン、Web、LLMの設定は保存しません。既定値を適用した設定を固定した順序へ変換し、SHA-256の指紋値を計算します。

## manifestの役割

`manifest.json`は公開中セグメントの一覧、generation、設定の指紋値、各コンポーネントのファイル種別、レコード数、
大きさ、SHA-256を保持します。セグメントは古い順から新しい順に並び、同じ文書IDは後のセグメントを優先します。
削除記録も同じ規則で古いレコードを隠します。

manifestは一時ファイルへ全量を書き、`fsync`後に不可分な名前変更で公開します。読み出し側はmanifestから参照される
すべてのコンポーネントを検証し、1件でも不一致ならスナップショット全体を開きません。

## コンポーネントの形式

永続化する整数はリトルエンディアンで保存し、C構造体やポインターをそのまま書きません。文字列はNULを含まない
長さ付きUTF-8です。各コンポーネントは共通ヘッダーを持ち、形式の版、ファイル種別、generation、レコード数、
ペイロードの大きさを検証します。

1つのコンポーネントのペイロード上限は256 MiBです。この上限は入力ファイル全体ではなく、直列化した個々のコンポーネントに適用します。
詳細な分割動作は[索引の更新と保守](index-lifecycle.md)を参照してください。

## 互換性

正式な`format_version`は2です。トークン分割、チャンク分割、ベクトルの`model_id`、`dimensions`、`metric`、
`filterable_fields`は指紋値へ含まれます。値が異なる索引とアプリケーション設定を組み合わせず、元文書から再索引してください。
