# 索引作成

この文書では、元文書を正式なNDJSONへ変換し、パッセージとベクトルを揃え、新しい索引を公開するまでの処理を説明します。コマンドの全オプションと正常出力は[コマンドリファレンス](command-reference.md)を参照してください。

## 処理の全体像

語彙索引は次の順序で作ります。

```text
元文書 → upsert NDJSON → UTF-8検証 → 字句分割とパッセージ分割 → セグメント作成 → マニフェスト公開
```

ベクトル対応索引では、チャンク結果と同じ順序で埋め込みを作る段階が加わります。

```text
元文書 → upsert NDJSON → パッセージ分割 → パッセージごとの埋め込み
      → `vectors`を`upsert`へ結合 → セグメントとANN作成 → マニフェスト公開
```

Yappod2のCコマンドは、`[embedding]`を読んで外部APIを呼びません。埋め込みはlocal-files、Wikipediaアダプター、または利用者側の処理で事前に生成します。

## 正式な文書NDJSON

ファイルはUTF-8で、空行を含めず、1行に一つのJSONオブジェクトを書きます。`build`は`upsert`だけを、`update`は`upsert`と`delete`を受理します。

```json
{"operation":"upsert","id":"doc-1","url":"https://example.test/1","title":"検索入門","body":"検索対象となる本文です。","metadata":{"language":"ja","year":2026},"updated_at_unix_ms":1784516400000}
```

| フィールド | 必須 | 型、既定値、上限 | 索引での扱い |
|---|---|---|---|
| `operation` | 必須 | `upsert` | 文書を追加または同じIDの新しい版として登録します。 |
| `id` | 必須 | 空でない文字列、最大255バイト | 文書の一意な識別子です。更新・削除・重複排除の単位です。 |
| `url` | 任意 | 文字列、`""`、最大8192バイト | 検索結果と引用情報へ返します。 |
| `title` | 任意 | 文字列、`""`、最大255バイト | 題名フィールドとして字句へ分割し、本文より高い重みで検索します。 |
| `body` | 必須 | 文字列、最大1048576バイト | 本文フィールドとして字句へ分割し、パッセージの分割元にも使います。 |
| `metadata` | 任意 | JSONオブジェクト、`{}`、最大1048576バイト | 文書コンポーネントへ保存し、設定したパスだけを絞り込み用索引へ保存します。現行の検索レスポンスには返しません。 |
| `updated_at_unix_ms` | 任意 | 0以上のJSON整数、`0` | 利用者側で管理する更新時刻です。順位計算には使いません。 |
| `vectors` | ベクトル有効時に必須 | 数値の二次元配列 | チャンク順に一つずつ対応するfloat32ベクトルです。 |

JSONオブジェクトの未知の最上位キーと、すべての階層の重複キーを拒否します。`metadata`はオブジェクトだけを許可し、キーをUTF-8バイト順へ並べた空白なしのJSONとして保存します。文字列にはNULを含められません。

`delete`は次の二つのフィールドだけを持ちます。

```json
{"operation":"delete","id":"doc-2"}
```

## トークンへの分割

`unicode_nfkc_casefold_v2`は、入力が正しいUTF-8であることを確認し、ICUのNFKC Casefoldで表記を正規化してから単語境界を求めます。これにより全角・半角や大文字・小文字など、Unicode正規化で同一視できる表記を同じトークンへ寄せます。

`title`、`body`、パッセージは別のフィールドとして字句へ分割します。トークンの出現位置は各フィールド内で0から数え、フレーズ検索に使用します。空白や句読点だけでトークンにならない部分は語彙辞書へ入りません。

トークナイザーIDは索引の互換条件です。別のトークナイザーで作った語彙や検索トークンを混ぜないため、変更時は索引を作り直します。

## パッセージへの分割

本文は`chunking.max_chars`以内のパッセージへ分けます。文字数はUTF-8バイト数ではなくUnicodeコードポイント数で数えます。

1. ICUの書記素境界から、`max_chars`以内で最も後ろの安全な境界を探します。
2. その範囲に文境界があれば、最後の文境界を優先します。
3. 次のパッセージは`overlap_chars`以下の範囲を重ねます。
4. 結合文字列や絵文字列の途中では分けません。

各パッセージは、文書ID、0始まりの`ordinal`、元本文の`start_char`と`end_char`、本文を持ちます。`start_char`と`end_char`は元本文のUnicodeコードポイント位置です。パッセージIDは文書IDと分割後の内容から決定的に生成されます。同じ文書ID、本文、分割設定なら同じIDになります。

`max_chars`は1〜1048576、`overlap_chars`は0以上かつ`max_chars`未満です。重なりを大きくすると境界付近の検索漏れを減らせますが、索引量、埋め込み回数、同じ内容の重複候補が増えます。

## `prepare`でチャンク結果を確認する

```sh
./build/yappo_makeindex prepare \
  --config application.toml \
  --input documents.ndjson \
  --output passages.ndjson
```

出力は1行1パッセージです。

```json
{"operation":"upsert","document_id":"doc-1","passage_id":"p-9a6d14af8d0bd5f4","ordinal":0,"start_char":0,"end_char":12,"text":"検索対象となる本文です。","metadata":{"language":"ja"}}
```

この中間形式は、文書とパッセージの対応を確認し、埋め込みアダプターへ入力するためのものです。`build`へ直接渡す文書形式ではありません。ベクトル付き文書NDJSONを作るときは、元の`upsert`へ`vectors`を追加します。

## 埋め込みと`vectors`

ベクトル対応索引では、一つの`upsert`に次のような二次元配列を入れます。

```json
{"operation":"upsert","id":"doc-1","title":"検索入門","body":"検索対象となる本文です。","metadata":{"language":"ja"},"vectors":[[0.12,-0.04,0.91],[0.08,0.33,0.72]]}
```

検証条件は次のとおりです。

- 外側の配列数が、その文書から現在の`[chunking]`で生成するパッセージ数と一致します。
- すべての内側配列の長さが`vector.dimensions`と一致します。
- すべての値がfloat32で表現できる有限値です。NaN、正負の無限大、float32範囲外を拒否します。
- パッセージ通し番号とベクトルの配列順を変えません。
- 索引作成と検索ベクトル生成に、同じ実モデル、プロンプトのプロファイル、前処理、次元数、距離尺度を使用します。

`vector.model_id`は、これらの互換条件を運用上識別する名前です。実装はベクトル値からモデルを推定しません。

## メタデータによる絞り込みに使う値

`metadata.filterable_fields`へ指定したJSONパスだけを`metadata.yap2`へ保存します。たとえば次の設定です。

```toml
[metadata]
filterable_fields = ["language", "author.country", "year"]
```

```json
{"language":"ja","author":{"country":"JP"},"year":2026,"internal_note":"索引化しません"}
```

この例では`language`、`author.country`、`year`を`filter`に使えます。元の`metadata`オブジェクト全体は`documents.yap2`に保存されます。ただし、現行の検索・取得レスポンスは`metadata`自体を返しません。`internal_note`は保存されますが、絞り込みにもレスポンスにも使われません。

null、真偽値、数値、文字列を保存します。対象が配列ならスカラー要素を複数値として保存します。オブジェクトはさらにパスを指定しない限り値として保存しません。

## 新規索引を作る

語彙索引の最小手順です。

```sh
./build/yappo_makeindex build \
  --config examples/config.lexical.toml \
  --input examples/documents.lexical.ndjson
```

`[index].directory`は存在していないパスでなければなりません。索引作成は同じ親ディレクトリに一時索引を作り、入力を最後まで処理し、全コンポーネントとマニフェストを検証した後だけ最終名へrenameします。

索引作成の入力総件数に100件の上限はありません。10000件ずつ内部処理単位へ分けて処理します。内部処理単位ごとに世代が進むため、大きい入力の初回索引作成でも最終世代は2に限りません。

## セグメントの分割計画

一つの`.yap2`ペイロードは最大256 MiBです。分割計画は文書とその全パッセージを一単位とし、次のコンポーネントサイズを事前に見積もって文書境界でセグメントを分けます。

- 文書
- 語彙、転置リスト、出現位置
- メタデータ
- ベクトル
- 削除標識

一つの文書だけでどれかの上限を超える場合、文書ID、必要量、コンポーネント名を報告して失敗します。文書の途中を別セグメントへ分けることはしません。

## 公開後に確認する

```sh
./build/yappo_makeindex verify --config application.toml
```

次も確認します。

```sh
./build/search \
  --config application.toml \
  --mode lexical \
  --query "確認用の語句"
```

ベクトル対応索引では、同じ次元数の既知ベクトルで`vector`と`hybrid`も確認します。`manifest.json`のコンポーネント一覧に`vectors.yap2`と`vectors.usearch`があり、索引`config.toml`のモデル識別子、次元数、距離尺度が意図した値であることを確認してください。

## 語彙索引をベクトル対応へ移行する

既存索引へベクトルコンポーネントだけを後付けするコマンドはありません。次の手順で別ディレクトリへ作り直します。

1. 元の文書NDJSONを保存します。
2. 新しいTOMLで`[chunking]`による文書分割と`[vector]`によるベクトル形式を確定します。
3. `prepare`またはアダプターでパッセージ順を確定します。
4. 同じ順序で埋め込みを生成し、`vectors`を元文書へ結合します。
5. 新しい`index.directory`へ`build`します。
6. `verify`、語彙検索、ベクトル検索、複合検索を確認します。
7. アプリケーション用TOMLの`index.directory`を切り替えます。

旧索引は、切替後の確認と必要な切り戻し期間が終わるまで保存してください。
